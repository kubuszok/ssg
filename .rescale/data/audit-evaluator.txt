================================================================================
 TOP 10 MISSING / SIMPLIFIED METHODS BY SPEC IMPACT
================================================================================

 1. _execute (MISSING)  ~100 lines
    Full module execution with environment save/restore. Required for correct
    @use/@forward module system. Current ad-hoc version skips save/restore of
    _root, _parent, _endOfImports, _extensionStore, _stylesheet, _mediaQueries,
    _declarationName, _inUnknownAtRule, _inKeyframes, _configuration, etc.
    Impact: modules can pollute each other's state.

 2. _combineCss (MISSING)  ~60 lines
    Topological sort and merge of module CSS trees. Required for correct output
    ordering when multiple modules are loaded. Also applies cross-module extensions.
    Impact: multi-module output ordering and extension propagation broken.

 3. _loadModule (MISSING)  ~80 lines
    Centralised module loading with cycle detection, configuration handling,
    stack frames. Current ad-hoc code in visitUseRule/visitForwardRule duplicates
    some logic but misses configuration assertion, module caching, cycle spans.
    Impact: configuration errors not caught, modules loaded multiple times.

 4. visitCssStyleRule / all CssVisitor methods (STUB)  ~250 lines total
    Required for evaluating CSS from @imported modules that themselves contain
    @use rules. Also used by meta.load-css().
    Impact: @import of modules with @use/@forward fails at runtime.

 5. _withParent with through: predicate (SIMPLIFIED)  ~30 lines
    Dart's _withParent + _addChild use a `through:` predicate to bubble at-rules
    and media rules past style rules. SSG's version lacks this, causing incorrect
    nesting of @media, @supports, unknown @-rules inside style rules.
    Impact: incorrect CSS output structure for nested at-rules.

 6. _visitCalculation / _visitCalculationExpression (SIMPLIFIED)  ~200 lines
    Missing: argument arity checks, whitespace around +/- validation, adjacent
    value checks, inLegacySassFunction back-compat, calc-size, unsimplified
    output in @supports context.
    Impact: spec failures for edge-case calculations.

 7. visitSelectorExpression (SIMPLIFIED)  ~5 lines
    Returns String instead of SassList. Dart returns the selector as a nested
    SassList structure suitable for list operations (nth, length, etc.).
    Impact: `& { }` works for interpolation but fails for list operations on &.

 8. meta constructor functions: get-function, call, get-mixin, load-css, apply
    (MISSING)  ~200 lines
    These are defined in the dart constructor and use closure state. SSG has
    some in MetaFunctions but the evaluator-dependent ones (get-function with
    _addExceptionSpan, call with SassFunction dispatch) are missing.
    Impact: meta.get-function(), meta.call(), meta.load-css() don't work.

 9. _withStackFrame / _stackTrace / _addExceptionSpan / _addExceptionTrace
    (MISSING)  ~100 lines
    Stack trace infrastructure for error reporting. Without this, runtime errors
    have no stack trace and span information is incomplete.
    Impact: poor error messages, no @debug/@warn traces.

10. _declarationName / nested declaration prefix (MISSING)  ~15 lines
    The `_declarationName` field enables `font: { family: sans; }` syntax where
    `font-family` is synthesized from the outer + inner names.
    Impact: nested declaration shorthand syntax broken.

================================================================================
 KEY OBSERVATIONS
================================================================================

 1. EXPRESSION EVALUATION is mostly complete. All expression visitor methods
    exist and produce correct results for the common cases. The main gaps are
    edge-case validation (plainCss guards, calculation arity, whitespace
    checks) and source-map/span tracking.

 2. STATEMENT EVALUATION is functionally present for single-file compilation.
    All statement visitors exist. The gaps are in multi-module orchestration
    (_execute, _combineCss, _loadModule) and CSS-nesting/bubbling fidelity
    (through: predicates, _inKeyframes, _declarationName prefix chain).

 3. MODULE SYSTEM (@use/@forward) works for basic cases but lacks the full
    Configuration / ExplicitConfiguration / _assertConfigurationIsEmpty
    machinery. The ad-hoc approach duplicates some logic and misses validation.

 4. CSS VISITOR is completely stubbed. This blocks @import of modules that
    themselves use @use/@forward (the _ImportedCssVisitor path).

 5. ERROR REPORTING lacks stack traces entirely. No _withStackFrame, no
    _stackTrace, no _addExceptionSpan. Errors get spans but no call stack.

 6. DEPRECATION WARNINGS are partially wired. Some specific deprecations
    (slash-div, mixed-decls, abs-percent) are emitted, but the general
    _warn(message, span, deprecation) 3-arg form with dedup and quietDeps
    is not implemented.

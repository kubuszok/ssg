ARCHITECTURAL DIFFERENCES
===============================================================================

1. EXPRESSION PARSER DESIGN
   dart-sass: Single _expression() method using operator-precedence climbing
   with closures (addOperator, addSingleExpression, resolveOperations) that
   consume directly from the scanner. Handles ALL expression forms including
   interpolation, bracket lists, selectors, unicode ranges, etc.

   ssg-sass: TWO parallel implementations:
   a) Text-based: _expression() collects raw text, then _parseSimpleExpression()
      does regex-like string parsing. Used for most contexts.
   b) Recursive-descent: _rdExpression() is a partial port of dart-sass's
      _expression(). Currently only wired for @return and variable declarations
      when the value looks like a single function call.

   Impact: The text-based path cannot handle nested interpolation, bracket
   lists, unicode ranges, or selector expressions correctly. The RD path
   lacks many expression forms (hash, selector, important, percent, unicode,
   modern if()).

2. CONTEXT FLAGS
   dart-sass tracks 9 parsing context flags:
   _inMixin, _inContentBlock, _inControlDirective, _inUnknownAtRule,
   _inPlainCssFunction, _inStyleRule, _inParentheses, _inExpression,
   _isUseAllowed

   ssg-sass: NONE of these flags are tracked. This means:
   - @use/@forward are accepted anywhere (should be root-only)
   - @function/@mixin inside mixin/control directives not rejected
   - Style rules inside @function not rejected
   - _inParentheses slash-separation reparse not triggered
   - plainCss mode has gaps in enforcement

3. INTERPOLATION MODEL
   dart-sass: InterpolationBuffer builds Interpolation objects with mixed
   String/Expression contents. singleInterpolation() returns (Expression,
   FileSpan). Used pervasively for selectors, identifiers, declaration values.

   ssg-sass: _parseInterpolatedString() does post-hoc text-based #{} detection.
   This misses nested interpolation, escape handling in interpolation context,
   and correct span tracking.

4. SELECTOR PARSING
   dart-sass: Full InterpolatedSelector AST with InterpolatedSelectorList,
   InterpolatedComplexSelector, InterpolatedCompoundSelector, and 10+ simple
   selector types. Used when _parseSelectors is true.

   ssg-sass: All 14 selector-parsing methods are MISSING. Selectors are
   raw text collected by styleRuleSelector() in ScssParser.

5. MEDIA QUERY PARSING
   dart-sass: 6 methods (_mediaQueryList, _mediaQuery, _mediaLogicSequence,
   _mediaOrInterp, _mediaInParens, _expressionUntilComparison) produce
   structured Interpolation with comparison operators.

   ssg-sass: All 6 methods MISSING. Media queries collected as raw text.

6. @SUPPORTS CONDITION PARSING
   dart-sass: 5 methods (_supportsCondition, _supportsConditionInParens,
   _supportsDeclarationValue, _trySupportsOperation) build structured AST
   (SupportsNegation, SupportsOperation, SupportsDeclaration, etc.)

   ssg-sass: All 5 methods MISSING. Conditions parsed as text into
   SupportsAnything or SupportsFunction heuristically.

===============================================================================
KEY FINDINGS
===============================================================================

1. The SSG parser is functional for basic SCSS: variables, declarations,
   style rules, most at-rules, and simple expressions work. The text-based
   expression parser handles common cases through string manipulation.

2. The partial RD expression parser (_rdExpression) is a good foundation but
   is only wired for a narrow case (variable declarations with function-call
   values). Completing and unifying this is the single highest-impact task.

3. The ENTIRE selector parsing subsystem (14 methods, ~300 lines) is absent.
   This blocks _parseSelectors=true mode needed for accurate selector
   manipulation.

4. Context flags are completely absent, meaning the parser does not enforce
   Sass scoping rules (e.g., no style rules inside @function, @use must be
   before other rules, etc.). This is a correctness issue but typically only
   matters for error cases.

5. The @while rule is entirely missing — a simple omission that could be
   fixed in <30 lines.

6. Modern CSS features (if() expression, bracket lists, unicode ranges) are
   all unimplemented.

7. The dual expression-parser architecture (text-based + RD) creates
   maintenance burden. The end goal should be to complete the RD parser
   and eliminate the text-based path entirely.
